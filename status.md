# Project status

Last updated: **2026-05-03**

## Where we are

We are **past initial scaffolding**: the monorepo runs Medusa v2, the official Next.js starter storefront, and backing services in Docker. **Phase 1 (foundation)** is complete, including **MinIO + Medusa S3 file module**. **Phase 4 (visual redesign)** is complete — see **`frontend.md`** for the finished NEXMART theme rollout. **Phase 2 is complete**: catalog translation coverage passes for the current seeded catalog (EN / AR / HE primary product/category fields), multi-store primitives and tenant-scoped runtime behavior are in place, and the verification/stabilization checks now pass. **Phase 3 product modeling is now complete**: specifications, preset-driven variant merchandising, workflow persistence, checkout/order continuity, analytics, store-level preset defaults, and PDP media enhancements are all in place. **Phase 5 is being stopped at its current local milestone**: the completed checkout/cart improvements stay in place, but the **remaining local Phase 5 follow-ups are intentionally skipped for now** because the last local steps are looping without enough forward progress. **Stripe-specific follow-ups remain deferred** until the project is running in a server-hosted, HTTPS-capable environment. **Phase 10 local payment closeout is now complete for the current scope**: PayPal is implemented and verified locally, and the former manual provider is now presented consistently as **Offline payment** rather than a generic test placeholder. The remaining payment follow-through is hosted PayPal webhook/public-HTTPS verification. **Phase 11 is now stopped at the current local milestone after six landed slices across backend and storefront**: anonymous store analytics ingest routes (`/store/analytics/preset`, `/store/analytics/whatsapp`) are rate-limited in Medusa middleware, those analytics counters now live in Redis with atomic expiry instead of process-local buckets, Medusa config rejects weak production JWT/cookie secrets instead of silently falling back to predictable defaults while validating auth/store/admin CORS env wiring at startup, storefront tenant/cache/locale metadata cookies now use hardened server-side attributes (`httpOnly`, `sameSite=strict`, `path=/`, `secure` in production), the custom storefront analytics POST proxies now reject missing-origin and cross-origin requests before forwarding to Medusa, and state-changing storefront server actions now enforce the same trusted-origin check across cart, checkout, customer, locale, order-transfer, and onboarding flows. The current local storefront CSRF/origin hardening pass is complete for the active write surfaces, and focused coverage now includes helper-level trusted-origin tests, an action-level customer mutation rejection test, and a runtime-facing Playwright replay that proves hostile Add to cart server-action requests are rejected. The remaining HTTPS and firewall items are deployment-side checks to revisit only after the server is deployed and started with its real HTTPS entrypoint. **Phase 12 is now started with a first catalog-translation tooling slice**: Medusa now has collection translation upsert and verification scripts, and the catalog translation audit now includes collections alongside products and categories, which closes a real multilingual workflow gap for collection-driven storefront surfaces. **Phase 6 is active** with the WhatsApp ordering, preview, note, analytics, admin, Playwright, and CI slices landed. **Phase 7.1 is complete for the current scope**: stock-mode behavior is aligned across PDP, cart, and browse cards; shopper-facing stock labels use the shared EN / AR / HE copy layer; `default_stock_mode` fallback is honored; admin control is in place; and fallback behavior is covered by shared/unit/component tests plus backend integration assertions. **Phase 8 is complete for the agreed core scope**: ACL role/permission matrix, authenticated admin context (`metadata.acl_role` + store assignments), expanded admin route middleware (catalog, orders, analytics, users, settings, regions/sales-channels read vs write, inventory, and API keys), **`api_keys.secrets` reserved for super_admin** with secret-key operation and list lockdown for other roles, shared HTTP ACL helpers (`apps/medusa/src/api/acl-http.ts`), **HTTP + unit tests**, **Admin → ACL Users** (`acl-user-roles` route) with `/admin/acl/user-roles` and a **current admin context** readout (`GET` and `POST` `/admin/acl/roles` require `users.manage`). **Phase 9 (admin panel roadmap track) is complete for the current scope**: **9.1** store overview is done, **9.2** product management is covered by **Catalog hub** plus Medusa product screens, and **9.3 / 9.4** are satisfied by native Medusa Dashboard flows unless a future custom requirement appears. **Phase 4.5 RTL/LTR + language-copy completion** remains in stabilization after the broader home/layout/PDP copy and QA pass.

## How to run (quick reference)

1. `docker compose up -d`
2. `pnpm dev:medusa` → API + admin UI (e.g. `http://localhost:9244/app`)
3. `pnpm dev:storefront` → storefront on **8000**
4. `pnpm --filter medusa run preset:analytics -- --days=7 --limit=10` → print preset analytics aggregates from Postgres
5. Admin → **Preset Analytics** (sidebar) → view dashboard with preset/product performance metrics, events, revenue
6. Admin → **Preset Defaults** (sidebar) → manage store-level preset combinations by product type
7. Admin → **ACL Users** (sidebar) → assign `acl_role` / `acl_store_ids` on admin users (`users.manage` required); `pnpm --filter medusa seed:acl-roles` seeds demo ACL identities

Copy `.env.example` at root as a reference; real secrets live in `apps/medusa/.env` and `apps/medusa-storefront/.env.local`.

## Decisions to remember

- **Traefik**: Ready as an external Docker network; when you containerize Medusa/Next, attach those services to `traefik` and add labels there.
- **DB.md vs Medusa**: Your custom schema is the **north star** for SaaS/i18n; Medusa already provides products, variants, orders, etc. Next step is to map features (nested categories, stock modes, translations) to **Medusa extensions** (modules, metadata, or separate services)—not to run `DB.md` as-is alongside Medusa without a migration strategy.

## Open items / caveats

- **Phase 10 framing**: Stripe is mostly covered locally and deferred for hosted HTTPS follow-up; PayPal and offline-payment checkout semantics are complete for the current local scope, so the remaining payment follow-through is hosted PayPal webhook/public-HTTPS verification.
- **Seed script**: May fail if regions already exist (`pnpm --filter medusa run seed`); use a fresh DB or adjust seeding if you need a repeatable demo reset.
- **Shipping seed mismatch (fixed in code, patched locally)**: older seeded databases can miss Israel checkout shipping because the `Middle East` region used `il` while the original fulfillment zone only covered Europe. Run `pnpm --filter medusa shipping:ensure-il` on older local DBs after pulling the latest seed/scripts.
- **Phase 5 stop-point (current local decision)**: the remaining local Phase 5 steps are intentionally skipped for now because the last slices are looping on the local machine without enough progress. Treat the current Phase 5 state as the local stopping point and continue with the next roadmap phase instead.
- **Stripe scope decision (current)**: no new Stripe work is active on the local machine for now. Resume Stripe-specific UI/webhook follow-ups only after deployment to a server-hosted HTTPS-capable environment.
- **Stripe webhook processing (local blocker)**: signed Stripe events can be delivered successfully to `/hooks/payment/pp_stripe_stripe`, but the current local dev flow is not yet reflecting the expected capture-state transition in Medusa after webhook delivery. Treat webhook validation as deferred rather than as the active blocking task.
- **Stripe payment-step card mount (local blocker)**: after Stripe payment-session initiation, the storefront now keeps the credit-card option visibly expanded, but the current local checkout tree still does not reliably remount into the actual Stripe card field. Treat this as deferred so the roadmap can keep moving.
- **Playwright local bootstrap**: local first-time browser install should use `pnpm test:e2e:install-browsers` (no sudo); CI uses `pnpm test:e2e:install-browsers:ci` inside `.github/workflows/checkout-deeplink-matrix.yml` when OS packages are needed.
- **Cart settings visibility (current local config)**: `/store/store-currency-config` currently returns `free_shipping_threshold: null`, so the free-shipping progress/CTA merchandising surfaces are intentionally hidden locally until that threshold is re-enabled.
- **Storefront peer warnings**: React 19 vs packages expecting React 18—monitor; align versions if you hit runtime issues.
- **Security**: Default JWT/cookie secrets in `.env` are for development only; rotate for any shared or production environment. Medusa now refuses weak/predictable production JWT/cookie secrets at config load, storefront tenant/cache/locale metadata cookies are now server-only with strict same-site scope, storefront analytics POST proxies and state-changing storefront server actions now enforce same-origin `Origin`/`Referer`, runtime-facing Playwright coverage now verifies hostile Add to cart server-action replays are rejected, and analytics rate limiting now uses Redis-backed shared counters with an in-memory fallback if Redis is temporarily unavailable.
